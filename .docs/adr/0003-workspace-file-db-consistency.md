# 0003. WebIDE 工作区：文件与数据库一致性策略

- 状态：Accepted
- 日期：2026-04-19
- 相关模块：apps/server（Workspace）、apps/ide

## 背景与问题

系统提供 WebIDE（OpenSumi）用于编辑页面源码工作区。存在两个事实载体：

- 数据库：page/component 等结构化存储（发布/访问依赖）
- 文件系统：工作区源码文件（IDE 编辑依赖）

需要保证：
- 用户在 IDE 中编辑 schema 文件后，数据库能及时同步；
- 首次打开 IDE 时，能从数据库生成工作区文件；
- 不引入“双源漂移”导致发布与源码不一致。

## 备选方案

### 方案 A：DB 为主，文件为投影（当前实现方向）

- 描述：
  - syncWorkspace：从 DB 生成 schema 文件到工作区；
  - writeWorkspaceFile：写文件；若写入的是 schema 文件，则同步回写 DB。
- 优点：
  - DB 仍是发布/访问的单一事实源；
  - IDE 编辑体验可用；
  - 逻辑简单、落地成本低。
- 缺点：
  - schema 文件与 DB 的一致性依赖于“识别 schema 文件路径”的规则；
  - 回写失败时可能造成短暂不一致，需要补偿机制。

### 方案 B：文件为主，发布时读取文件并解析入库

- 优点：IDE 与源码一致性天然更强。
- 缺点：发布链路变长；需要解析与校验；服务端需要访问工作区文件且要处理并发写入。

## 决策

选择方案 A：DB 为主，文件为投影，并在写 schema 文件时回写 DB。

补充要求（建议）：
- schema 文件写入采用“临时文件 + 原子替换”，降低写入中断风险；
- 回写 DB 失败时返回明确错误，并提示用户重试；必要时记录同步失败事件用于排查；
- 对 `path` 做严格校验，防止路径穿越。

## 影响与后果

- 可观测性：需要为“同步/回写失败”添加日志与指标。
- 安全：工作区文件读写必须鉴权，并对 path 做白名单约束。
- 演进：未来如需支持多文件构建/编译，可引入构建服务或队列，避免阻塞请求链路。
