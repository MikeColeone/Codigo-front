# 0001. 后端采用模块化单体（NestJS）

- 状态：Accepted
- 日期：2026-04-19
- 相关模块：apps/server

## 背景与问题

系统需要同时支持：

- REST API（鉴权、发布、模板、资源、后台治理）
- WebSocket（实时协同）
- 以 MySQL 为主的数据一致性写入（发布时 page/component/version 强一致）

在当前业务阶段，需要优先保证“开发效率、统一治理、可演进拆分”的平衡。

## 备选方案

### 方案 A：模块化单体（Modular Monolith）

- 描述：单 NestJS 应用进程，按业务域拆分 Module/Controller/Service。
- 优点：
  - 鉴权、异常、响应格式、日志等横切能力统一收口；
  - 事务一致性更容易实现；
  - 部署与运维复杂度低。
- 缺点：
  - 模块边界仅靠工程约束，运行时隔离弱；
  - 规模增长后可能出现耦合膨胀，需要持续治理。

### 方案 B：微服务（按域拆分服务）

- 描述：Auth/Page/Workspace/Collaboration/Admin 等拆分独立服务，通过网关聚合。
- 优点：独立扩缩容、隔离故障域、团队并行更强。
- 缺点：
  - 分布式事务与一致性复杂；
  - 服务治理（注册发现、配置中心、链路追踪）与运维成本显著提升；
  - 对当前阶段交付效率不友好。

## 决策

选择方案 A：模块化单体。

理由：
- 当前系统的核心写链路（发布/版本）需要事务一致性；
- 需要统一鉴权与后台治理能力；
- 保留未来拆分空间：通过模块边界、接口契约、依赖方向约束为拆分铺路。

## 影响与后果

- 对模块边界：必须持续按域拆分模块，避免出现“超级模块”。
- 对可扩展性：当协作（WebSocket）扩容需求上来时，需引入 Redis adapter 或拆分协作网关。
- 对数据库：生产环境需迁移到 migrations 与显式索引/约束，避免 synchronize 带来的隐式变更。
