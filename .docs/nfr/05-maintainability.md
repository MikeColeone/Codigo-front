# 9.4 可扩展性与可维护性设计

## 9.4.1 插件机制（可扩展性）

系统在 monorepo 中已具备插件/物料的基础设施：

- `@codigo/plugin-system`：插件注册协议（建议作为扩展点的“抽象层”）
- `@codigo/materials`：运行时 React 物料与 registry
- `@codigo/schema`：跨端类型/协议单一事实源

扩展建议：
- 插件能力以“事件编排（Flow）”为中心：组件事件触发动作链路执行。
- 扩展点必须先定义协议（schema），再实现运行时（materials/client/server），避免双源。

## 9.4.2 配置中心与环境隔离

建议配置分层：

- **Build-time 配置**：前端构建时注入（API_BASE_URL、feature flags）
- **Run-time 配置**：后端环境变量注入（JWT_SECRET、DB/Redis、第三方密钥）
- **动态配置（可选）**：灰度开关、限流策略等（可落 Redis/DB，并带审计）

## 9.4.3 发布策略（蓝绿 / 灰度）

### 前端

- 静态资源版本化（hash 文件名），配合 CDN 缓存与回滚。
- release（Next.js）建议使用蓝绿或滚动升级，确保 SSR 服务平滑切换。

### 后端

- 无状态化后可滚动升级；
- 对 WebSocket 连接，需要考虑升级时的连接迁移与重连策略；
- 重要接口变更采用兼容策略（新增字段可选、保留旧字段一段时间）。

## 9.4.4 回滚策略

- **代码回滚**：以 tag/版本为基线，回滚到上一个稳定版本。
- **数据库回滚**：
  - 推荐使用 migrations 并支持 down/rollback；
  - 对不可逆变更（删除列/改类型）必须先双写/灰度，再最终切换并清理。
- **数据修复**：对 page/component/page_version 的数据一致性提供脚本（导出/修复/回填）并记录执行审计。

## 9.4.5 可维护性：模块边界与单一职责

- `apps/*` 与 `packages/*` 的依赖方向是硬约束；跨域复用优先下沉到 `packages/*`。
- shared 只能承接基础设施，禁止承接单一业务域逻辑（避免横向穿透）。
- 对 Legacy 接口持续治理，避免重复能力长期共存。

## 9.4.6 文档与实现一致性的长期机制

- 任何接口/字段/模块边界变更必须同步更新 `.docs`：
  - 接口：更新 `modules/02-api-rest.md` 与相关时序图；
  - 数据：更新 `data/02-er-and-schema.md` 与数据字典；
  - 关键决策：新增/更新 ADR。
- 通过 CI 门禁自动提醒（见根脚本 `changes:gate` 与 `docs:check`）。
