# E2E 自动化测试（设计一致性验证）

目标：在测试环境中，以“设计文档”为基准编写端到端用例，验证文档与实现一致；通过率 ≥ 95% 作为评审验收门槛之一。

## 1. 工具选型（建议）

- 前端 E2E：Playwright（优先）或 Cypress
- API E2E：supertest（Nest 侧）或 Playwright 的 APIRequest
- 数据准备：测试库（独立 MySQL schema）+ 预置账号

## 2. 用例范围（与用例规约对齐）

对应 `.docs/use-cases/01-use-case-model.md`：

- 注册/登录（密码/短信验证码）
- 创建/编辑页面（最小可用：添加组件、保存状态）
- 发布页面（生成 pageId）
- 匿名访问 public 页面
- private 页面访问控制（401/403）
- 协作者管理（邀请/移除/改角色）
- 后台治理（用户冻结/解冻、角色变更、权限点校验）
- 资源上传（上传成功并可在列表中看到）
- 模板列表与详情（管理员 CRUD 可选）

## 3. 验收标准

- 用例通过率 ≥ 95%
- 覆盖关键错误分支（401/403/404/400）并断言错误码与响应结构
- 关键字段断言与数据字典一致（如 `visibility/expire_at/page_id`）

## 4. 测试数据与隔离

- 所有测试数据必须可重复创建与清理；
- 不依赖线上数据；
- OSS/短信服务在测试环境建议使用 stub/mock 或沙箱环境。
