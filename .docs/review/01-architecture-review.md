# 架构评审与验收流程

## 1. 评审会议组织（建议）

参与角色：
- 技术负责人（Owner）
- DBA
- 安全负责人
- 运维（SRE/DevOps）
- 测试负责人（QA）

输入材料（会前至少 1 天发出）：
- `.docs/README.md`（阅读顺序）
- `architecture/*`、`modules/*`、`data/*`、`nfr/*`
- ADR 草案（Proposed）

会议产出：
- 评审结论：通过/有条件通过/不通过
- 评审意见清单：问题、风险、改进建议、责任人、截止条件
- ADR 状态更新：Proposed → Accepted / Superseded

## 2. 评审检查清单（摘录）

- 架构图与部署拓扑可自洽（前后端/中间件/DB/第三方）。
- 接口清单完整，Legacy 有下线计划。
- 数据模型字段与代码一致，索引/约束合理。
- 安全：密钥不硬编码；鉴权/授权链路完整；日志脱敏。
- 可观测性：关键指标与告警规则可落地。
- 可维护性：模块边界清晰，依赖方向符合约束。

## 3. 评审记录模板（建议写入 ADR 或附录）

- 时间：
- 参与人：
- 结论：
- 意见与行动项：
  - [ ] 问题：
    - 风险等级：
    - 责任人：
    - 验收标准：

## 4. 设计基线与版本回溯

- 评审通过后：
  - 将 `.docs` 目录纳入版本库；
  - 在仓库层面打 tag 标记“设计基线版本”（tag 命名建议：`docs-baseline-YYYYMMDD`）；
  - 后续任何迭代都能追溯到对应设计基线。

## 5. 文档与实现一致性验收

- CI 门禁必须通过：
  - `pnpm docs:check`
  - `pnpm changes:gate`（PR 场景）
- 测试环境：
  - 基于设计文档编写端到端（E2E）用例，通过率 ≥ 95%
